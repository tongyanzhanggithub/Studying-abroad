# 上云之后的安全性 —— 运维清单

代码层的安全(会话、密码、限流、越权、SSRF、响应头)已经在应用里做完了,
清单见 `deploy/README.md` 和源码注释。**这份文档只讲代码管不到的那一层** ——
云平台配置和运维制度。这一层配错,应用写得再干净也没用:安全组开了 5432、
或者拿 root 账号的 AccessKey 到处用,数据一样会泄。

> 这不是一次性任务。分三档:**P0 = 接真实用户前必做**,
> **P1 = 上线第一个月内**,**P2 = 持续**。
> 涉及等保/隐私合规的部分是方向性提示,**最终以专业顾问/律师意见为准**。

---

## P0 —— 接真实用户之前必须做完

### 1. 安全组:只开该开的端口

这是最容易出事、后果最严重的一条。云服务器默认安全组常常是放开的。

| 端口 | 对谁开 | 说明 |
|---|---|---|
| 443 | `0.0.0.0/0` | HTTPS,对公网 |
| 80 | `0.0.0.0/0` | 只用于 80→443 跳转和 certbot 验证 |
| 22 | **只对你自己的固定 IP** | SSH,绝不对 `0.0.0.0/0` |
| 5432 | **谁都不开** | Postgres 只走本机 `127.0.0.1`,永远不进安全组 |
| 3000 | **谁都不开** | Next.js 只监听 `127.0.0.1`,由 nginx 反代 |

- 阿里云控制台 → ECS → 安全组 → 入方向规则,逐条核对,删掉任何 `0.0.0.0/0` + 全端口的规则。
- **验证**(从你自己机器):
  ```bash
  nmap -Pn 你的公网IP        # 期望只看到 80、443 open
  nc -zv 你的公网IP 5432     # 期望 timeout / refused,绝不能 succeeded
  ```
- 数据库确实只听本机:
  ```bash
  sudo ss -tlnp | grep 5432   # 期望 127.0.0.1:5432,不是 0.0.0.0:5432
  ```
  若是 `0.0.0.0`,改 `postgresql.conf` 的 `listen_addresses = 'localhost'` 后重启。

### 2. SSH 加固:关掉密码登录

自建服务器被撞库爆破 SSH 是最常见的入侵方式。

```bash
# 先确认你的公钥已经能免密登录,再改下面这些,否则会把自己关在外面
sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo sed -i 's/^#\?PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
sudo sed -i 's/^#\?PubkeyAuthentication.*/PubkeyAuthentication yes/' /etc/ssh/sshd_config
sudo systemctl restart sshd

# fail2ban:自动封反复失败的 IP
sudo apt install -y fail2ban
sudo systemctl enable --now fail2ban
```

- 用**密钥**登录,不用密码。私钥不出你自己的电脑。
- 有条件把 22 端口 SSH 收到堡垒机 / VPN 后面,安全组只放堡垒机 IP。

### 3. 不要用主账号 AccessKey —— 建 RAM 子账号

这条针对 OSS、短信这些要 AccessKey 的服务。

- **主账号(root)的 AccessKey 一律不生成、不使用。** 一旦泄露 = 整个阿里云账号沦陷。
- 给应用建一个 **RAM 子用户**,只授它真正要用的权限:
  - OSS:只授权那**一个** bucket 的读写(`oss:PutObject`/`GetObject`/`DeleteObject`),不是 `AliyunOSSFullAccess`。
  - 短信:只授 `AliyunDysmsFullAccess`(或更细)。
- 主账号登录**开启 MFA**(手机/硬件令牌)。
- AccessKey 只以环境变量进 `.env`,`.env` 权限 600,不进 git(`.gitignore` 已含)。

### 4. OSS 桶:私有 + 加密 + 签名访问

护照、身份证、学位证要迁到 OSS(见 `deploy/README.md`「这些文件是敏感个人信息」)。
迁的时候桶必须这样配:

- **读写权限 = 私有**(绝不是「公共读」)。公共读的桶 = 任何人凭 URL 就能下别人的护照。
- **服务端加密 = 开启**(OSS 的 SSE,或 KMS 托管密钥)。落盘即加密。
- 应用里**永远不返回永久公开 URL**,一律用**带过期时间的签名 URL**(几分钟有效),
  和现在 `/api/materials/[id]/file` 的归属校验配合。
- 关闭 bucket 的公网访问(如业务允许,只对 ECS 内网 endpoint 开放)。
- 开启**版本控制 + 删除保护**,防误删/勒索。

> 代码侧:存储层现在是 `local` provider。切 OSS 时新增 `oss` provider,
> 上传后只存 objectKey,下载时现签 URL —— 不要把 URL 存库。

### 5. HTTPS 与证书(README 已有,核对即可)

- `certbot --nginx` 配好,80→443 强制跳转。
- 只保留 **TLS 1.2 / 1.3**,关掉 1.0/1.1。
- HSTS 响应头已在 `next.config.mjs` 配好,证书上线当天自动生效。
- **证书自动续期**:`certbot renew` 的 systemd timer 默认已装,`sudo certbot renew --dry-run` 验证一次。

### 6. `.env` 与密钥

- `chmod 600 .env`,owner 是跑应用的那个用户。
- 上线前**重新生成** `AUTH_SECRET` 和 `CRON_SECRET`(各 `openssl rand -base64 32`),
  不要沿用任何出现在聊天/文档/git 历史里的值。
- 数据库密码换成强随机,不用 `.env.example` 里的 `compass_dev_pwd`。

## P1 —— 上线第一个月内

### 7. 云 WAF + DDoS 兜底

应用层限流(验证码 IP 闸、登录锁定)挡的是业务滥用,挡不住网络层洪水。

- 前面挂一层**阿里云 WAF**:防 SQL 注入/XSS 的通用规则、CC 攻击防护(比应用层限流更靠前)、
  可按地域/频率封禁。
- 阿里云默认有**基础 DDoS 防护**;预算允许上**DDoS 高防**。
- WAF 上配一条:`/admin`、`/advisor` 只对办公固定 IP 放行(后台没必要对全网开)。

### 8. 监控 + 告警(现在是盲的)

出了事要能第一时间知道,而不是等用户投诉。

- **健康检查告警**:`/api/health` 已就绪。用云监控/Uptime 每分钟探一次,
  非 200 立刻短信/钉钉告警。
- **资源告警**:CPU、内存、**磁盘使用率**(自建 Postgres + uploads 最容易撑爆磁盘)、
  数据库连接数。
- **安全告警**:后台连续登录失败(代码已记 `failedAttempts`/`lockedUntil`,
  接一个查询定时上报)、SSH 登录成功通知。
- **cron 失败告警**:两个定时任务失败会返回 500,日志在 `/var/log/compass-cron.log`,
  接进日志告警。
- 应用日志接**阿里云日志服务(SLS)**,别只留在本机 —— 机器挂了日志也没了,事后无法追溯。

### 9. 备份要能恢复,且异地

README 里的每日 `pg_dump` + `uploads` 打包是基础,补两点:

- 备份**同步到 OSS**(另一个 region 更好),留在同一台机器只防误删、不防机器挂。
- 备份文件里含明文材料 → **同步时加密**(客户端加密或 OSS SSE)。
- **每月真的做一次恢复演练** —— 没验证过的备份等于没有备份。
- OSS 侧备份桶开**版本控制**,防勒索软件把备份也一起加密掉。

### 10. 系统与依赖更新

```bash
# 自动装安全补丁
sudo apt install -y unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades
```

- 定期 `npm audit`,盯高危依赖(尤其 next / prisma / jose)。
- 关注 Next.js 安全公告,及时升小版本。

## P2 —— 长期制度

### 11. 合规(处理护照/身份证,这条绕不过)

平台收集、存储身份证/护照属**个人敏感信息**,除 ICP 备案外通常还涉及:

- **等保 2.0(网络安全等级保护)**:处理公民个人敏感信息的系统,一般需**二级等保**
  测评备案。走流程要时间,尽早启动。
- **《个人信息保护法(PIPL)》义务**:告知-同意(已在注册流程留痕)、最小必要、
  可删除/可导出、数据出境评估(PRD 10.7 要求数据留在境内,别踩这条)。
- **敏感信息生命周期**:定明确的**留存期限**和**到期删除**机制 —— 学生完成申请后,
  护照扫描件不该无限期留着。这是代码要配合做的(定时清理 + 硬删除)。
- 这些的具体适用范围**请以专业合规顾问/律师意见为准**,上面只是方向。

### 12. 访问审计与最小权限

- 运营后台的敏感操作(改价、看材料、导出线索)应留**操作审计日志**(谁、何时、对谁)。
- 定期复查 RAM 子账号权限、后台账号列表,离职/换人及时禁用(`/admin/accounts` 已支持停用)。
- 数据库给应用用的账号不要用超级用户,按需授权。

### 13. 事件响应预案

写一页纸就够,但要提前写:

- 发现疑似泄露/入侵时:先隔离(安全组封端口)、再取证(别急着重装)、
  轮换所有密钥(AUTH_SECRET/CRON_SECRET/DB 密码/AccessKey)、评估是否触发
  PIPL 的**告知义务**(向受影响用户和监管报告)。
- 平时就准备好:AccessKey 快速吊销入口、数据库快速改密脚本、联系人清单。
