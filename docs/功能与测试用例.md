# Compass 功能详情与测试用例

面向**手工验收测试**。每条用例给出:操作步骤 → 预期结果。
预期结果写得很具体,是为了让「差不多对」和「真的对」能被区分开。

> 测试环境:`http://8.208.125.209`(演示服)或 `http://localhost:3000`(本地)
>
> 🔴 标记 = **合规/安全红线**,失败必须停下修,不能放行。

---

## 〇、测试前准备

| # | 操作 | 预期 |
|---|---|---|
| P1 | 服务器上 `curl -s localhost:3000/api/health` | 返回 `{"ok":true}`。**这条会真的查数据库**,首页 200 不代表数据库通 |
| P2 | 建后台账号:`ADMIN_EMAIL=... npm run admin:create` | 打印邮箱 + 密码(只打印一次) |
| P3 | 建学生账号:`PHONE=13800138000 PASSWORD='至少12位' npm run demo:seed` | 打印手机号 + 密码,并显示「选校单 5 所 / 材料清单 N 项」 |
| P4 | 登录后台 → 「地区开放」→ 打开**英国** | **不做这步前台永远是空的**,是设计如此(数据没核对完不放出去) |

**演示环境已知限制**(以下不可测,不是 bug):
- 真实支付(mock,点确认即视为付款成功)
- 真实短信(需 `ALLOW_MOCK_SMS=true`,验证码只进服务端日志)
- 真实 LLM(mock,AI 返回固定占位内容)
- 对象存储(材料存本机磁盘,已 AES 加密)

## 八、安全与权限 🔴

| # | 操作 | 预期 |
|---|---|---|
| S1 | `curl -I http://<站点>/` | 含 `X-Frame-Options: DENY`、`Content-Security-Policy: frame-ancestors 'none'`、`X-Content-Type-Options: nosniff` |
| S2 | 把站点嵌进别人页面的 iframe | **无法嵌入**(防点击劫持) |
| S3 | 打开 `/robots.txt` | 屏蔽 `/app/` `/admin/` `/advisor/` `/assess/result/` `/api/` `/r/` |
| S4 | 未登录访问 `/app/dashboard` | 跳登录页 |
| S5 | 学生账号访问 `/admin` | 进不去 |
| S6 | 访问 `/api/dev/verify-collect`(生产) | **404**(开发自检路由生产不可达) |
| S7 | 服务器外部 `nc -zv <IP> 5432` | **连不上**(数据库绝不能对公网开放) |
| S8 | 同一 IP 用不同手机号连发验证码 | 第 21 次被拦(防短信轰炸/烧钱) |
| S9 | 查数据库 `verification_codes` 表 | `code` 字段是 **HMAC 摘要,不是明文** |
| S10 | 查 `users` 表 | `password_hash` 是 `scrypt$...`,**无明文密码** |

## 九、合规红线总检查 🔴

**这一节任何一条不通过,都不能对外接真实用户。**

| # | 检查 | 要求 |
|---|---|---|
| R1 | 全站文案搜「保录」「保offer」「100%」「包过」 | 无承诺录取的表述 |
| R2 | 所有概率数字 | 标为**预估区间** + 免责声明 |
| R3 | 文书模块 | **不存在**「一键生成全文」;协议里明确写不代写 |
| R4 | 未人工核对的数据 | UI 上有明确标注,不当确定值 |
| R5 | 退款规则 | **付款前**在页面上可见 |
| R6 | 注册流程 | 强制阅读并同意协议,同意时间与版本号入库 |
| R7 | 数据权利 | 可导出、可注销删除 |
| R8 | 手机号用途 | 页面明示「仅用于登录与申请提醒,不向第三方出售」 |
| R9 | 学生材料 | 落盘加密;取文件校验归属 |
| R10 | AI 使用政策 | 零容忍院校在文书页有显著警告 |
